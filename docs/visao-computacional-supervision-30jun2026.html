Visão computacional ajuda —
mas no travamento do alvo,
não nas estatísticas.

Conclusão honesta: sim, há aplicabilidade real, porém concentrada e com custo arquitetural alto. O supervision não é um pip install que melhora o produto de hoje — é construir um sub‑pipeline de visão computacional. O encaixe mais forte não é "consertar a contagem de estatística" (esse modo já está desligado); é blindar a seleção do atleta‑alvo — exatamente a dor que os últimos 5 commits da "Quadra Interativa" atacam na mão.

a recomendação em uma frase

Não adote em bloco. Faça um spike cirúrgico da hipótese de maior valor (travar o alvo) num único clipe, offline, num ambiente com GPU — zero treino de modelo (pessoa é classe pronta). Se provar que mata a falha do "atleta errado", aí decide o investimento em infra assíncrona. Homografia é fase 2; bola/velocidade é pesquisa; bounce/in‑out se descarta.

02

O que o supervision é — e o que não é

O ponto que muda toda a conversa de custo. Verificado nas fontes primárias (README, código‑fonte e docs), não de memória.

supervision = caixa de ferramentas

O próprio README: "model agnostic. Just plug in any classification, detection, or segmentation model." Ele não faz inferência e não faz chamada de rede — é numpy + opencv puro. É o "pandas + matplotlib da detecção de objetos": padroniza a saída de qualquer modelo num sv.Detections e te dá tracking, zonas, anotadores e métricas. Você traz o modelo.

roboflow/sports = exemplos (futebol e basquete)

Repo de receitas aplicadas, não biblioteca de produção. Não tem tênis — nada de beach tennis. O reutilizável de lá é genérico: o ViewTransformer (homografia via cv2.findHomography, agnóstico de esporte) e a renderização "radar" (draw_pitch). O SoccerPitchConfiguration é hardcoded — você escreveria um BeachTennisCourtConfiguration à mão.

leitura honesta do que "adotar supervision" significa

Não é instalar uma lib e ganhar análise de tênis. É escrever um pipeline de CV (detector → tracking → homografia → lógica), usando o supervision como cola de alta qualidade. Os primitivos certos existem e são bons — sv.ByteTrack, sv.PolygonZone, sv.InferenceSlicer (bola pequena), sv.KeyPoints, anotadores, get_video_frames_generator/VideoSink — mas a montagem e os modelos são por sua conta. Licença MIT, Python ≥3.10 (já é o seu caso). O supervision não puxa torch; o peso de 1–3 GB vem do backend de modelo (Ultralytics / Inference).

03

O estado atual — por que o encaixe não é onde parece

Três fatos do código de hoje que reposicionam a decisão. Todos verificados em app/tennis/.

1 · 100% "manda pro Gemini"

Não existe CV nenhum hoje: zero OpenCV, numpy, torch, extração de frame, tracking ou homografia. O vídeo sobe como bytes opacos pro Files API; VideoMetadata(fps) só diz ao Google como amostrar. É greenfield — nada pra refatorar, mas nada pra reusar.

2 · o modo match está desligado

routing.py:decide_mode sempre retorna "clip" (linhas 83‑96). Todo o MatchAnalysis (aces, winners, rally length, % de saque), os pesos por gênero e os benchmarks ATP/WTA são dead code na requisição. O gargalo "contar estatística por VLM é aproximado" foi contornado, não resolvido.

3 · o schema recusa métrica de propósito

As zonas são qualitativas (fundo/meio/rede), nunca coordenadas. O prompt proíbe o modelo de dar velocidade/tempo em número. A justificativa está escrita no código: "não dá medida métrica sem homografia".

o gap está escrito no próprio código — não é encaixe forçado

O repo documenta exatamente a fronteira que o supervision foi feito pra cruzar:

"Zonas qualitativas e relativas (NÃO coordenadas): o VLM raciocina espaço grosseiro a partir de uma câmera, mas não dá medida métrica sem homografia."app/tennis/models.py:60‑62
"NÃO estime tempos de reação nem velocidades em números (ms, m/s) — descreva qualitativamente (rápido/lento/tardio)."app/tennis/prompts.py:184‑185

Homografia + tracking é o núcleo do supervision (ViewTransformer, ByteTrack, PolygonZone). E mais: a feature recém‑commitada "Quadra Interativa" — usuário escolhe a câmera, toca o quadrante onde o alvo começa e descreve a cor pra "seguir o mesmo atleta" — é um substituto manual e humano de detecção de quadra + tracking de jogadores. Os seams do schema (camera_reference, target_quadrant, target_appearance, e o auto‑check _check_target_sector em service.py:310‑341) já estão prontos pra um módulo de CV preencher automaticamente.

04

A divisão que decide tudo — quantitativo × qualitativo

VLM e CV são fortes em coisas opostas. Sobrepor os dois no lugar errado é desperdício; no lugar certo, é complementaridade.

VLM (Gemini) — bom em PERCEPÇÃO e COACHING

  • Notas técnicas 0‑10 (preparação, contato, biomecânica, base).
  • Classificação de golpe / fase / lateralidade.
  • Narrativa, evidência visual, leitura tática relacional do ponto.
  • Incerteza auto‑declarada (phase_confidence…).

▸ é o clip de hoje, e funciona. CV não melhora isto.

CV (supervision + modelo) — bom em MEDIÇÃO e GEOMETRIA

  • Contagens exatas (golpes, trocas) e identidade estável por frame.
  • Posição na quadra, distância, heatmap, ocupação de zona.
  • Velocidade (px→metros via homografia) — quando a bola é detectável.
  • Gabarito determinístico pra validar/calibrar.

▸ é o quantitativo que o schema recusa e o match desligado.

frase estratégica

CV é a resposta durável para medição — números determinísticos que o VLM sempre vai aproximar mal. O VLM continua dono da percepção e do coaching. O supervision é a cola certa pra metade‑medição — desde que vocês topem deixar de ser "uma chamada de API" e virar "um pipeline de vídeo".

05

Onde encaixa aqui — 3 superfícies, por ROI

Ordenadas por valor/esforço. (A) ataca a dor ativa e não precisa de modelo treinado; (B) destrava métrica nova reusando o que a UI já coleta; (C) é brinde de baixo risco.

ATravar o atleta‑alvo por detecção + tracking maior alavancasem treinar modelo

É a recomendação principal: ataca a dor ativa (os 5 commits da Quadra Interativa) e os seams já existem.

✕ HOJE

Com 4 jogadores, ancoragem por aparência ("o de azul") é fraca. O usuário escolhe câmera + toca quadrante + descreve cor; o _check_target_sector só consegue avisar "pode ter pego o atleta errado". subject_lock_confidence é auto‑declarado pelo VLM.

✓ COM CV

Detecção de pessoa (YOLO classe "person" — pré‑treinada, zero treino) + sv.ByteTrack dá IDs estáveis aos 4 jogadores. O usuário clica no jogador de verdade; o tracker segue ao longo do clipe. Daí ou (i) destaca/recorta o alvo nos frames e manda esse vídeo anotado pro Gemini — ele fisicamente não analisa a pessoa errada — ou (ii) preenche o lock de forma determinística.

Honestidade: em duplas, quadra pequena = oclusão → o ByteTrack troca IDs quando jogadores se cruzam. Mitigável (re‑ID por aparência, que vocês já coletam), mas não é mágica.

BDestravar métrica espacial REAL via homografia fase 2

06

Ressalva crítica — é beach tennis, não tênis

A literatura de "tênis‑CV" assume quadra de saibro/dura, singles, com quique. O seu schema é beach tennis, duplas (models.py:81‑83). Isso derruba metade do toolkit clássico.

o que NÃO se aplica
  • Bounce / in‑out. Beach tennis é jogo de voleio (sem quique no chão). Toda a técnica de "detecção de bounce" e "linha in/out" — staple do tênis‑CV — é irrelevante aqui.
  • Modelos prontos não transferem. Os modelos de keypoints/bola de tênis do Roboflow Universe foram treinados em quadra dura, saque‑e‑fundo. Areia, rede a 1,7m, quadra 16×8m, 4 jogadores aglomerados são outro domínio.
o que SOBREVIVE
  • Detecção/tracking de pessoas — classe "person" é genérica, funciona livre. É a superfície (A).
  • Heatmap / ocupação de posição via homografia com cantos manuais (B).
  • Contagem de golpes por contato raquete‑bola (se a bola for detectável).
  • Identidade do alvo — o de maior valor, e o mais robusto.
consequência prática

A bola é o calcanhar: bola de beach tennis é pequena, rápida, com motion blur, contra areia/céu. Detectá‑la exige modelo treinado em dado próprio (o supervision tem o InferenceSlicer/SAHI pra ajudar com objeto pequeno, mas não inventa o detector). Por isso: lidere por métrica de jogador (só precisa de "person" + homografia) e trate velocidade de bola como pesquisa, não como base. Descarte bounce/in‑out.

07

O custo real — sem maquiagem

CV de verdade colide de frente com a postura atual (uma chamada de API, síncrona, VPS única CPU‑only que é o host de produção).

1 · colisão arquitetural

Hoje é "manda bytes pro Gemini". Inferência por‑frame numa VPS CPU‑only, host de produção, síncrona, sem fila de jobs, força exatamente a migração pra assíncrono (job + polling) que o blueprint §09 já aponta — mais GPU local ou inferência hospedada.

2 · peso & deps

O supervision em si é leve (numpy/opencv). O peso de 1–3 GB vem do backend de modelo (torch + Ultralytics/Inference) vs. a imagem enxuta de hoje (só google-genai + FastAPI + psycopg).

Pra vídeo, o padrão realista é inferência local em GPU (com o supervision pra tracking/zonas/anotação), reservando a API hospedada pra protótipo ou volume baixo.

08

O spike — testar a hipótese de maior valor barato

Antes de mexer em qualquer infra: provar (ou refutar) a superfície (A) num único clipe, offline, num ambiente com GPU (Colab serve). Zero treino de modelo. Esqueleto abaixo — ponto de partida, não cópia‑cola garantida.

# spike_target_lock.py — prova de conceito (A): travar o atleta-alvo
# Rodar com GPU (Colab serve). NÃO entra em produção — é experimento offline.
#   pip install supervision ultralytics
import numpy as np, supervision as sv
from ultralytics import YOLO

SOURCE = "00000201.mp4"            # o clipe de calibragem que vocês já usam
TARGET = "clip_alvo_anotado.mp4"   # saída: mesmo clipe, só o ALVO destacado

model   = YOLO("yolo11m.pt")         # classe 0 = 'person' (pré-treinado, zero treino)
tracker = sv.ByteTrack()
box, trace, label = sv.BoxAnnotator(), sv.TraceAnnotator(), sv.LabelAnnotator()
info    = sv.VideoInfo.from_video_path(SOURCE)
TARGET_ID = None                       # trava no 1º frame, a partir do quadrante da UI

def annotate(frame, i):
    global TARGET_ID
    res = model(frame, classes=[0], verbose=False)[0]
    det = sv.Detections.from_ultralytics(res)
    det = tracker.update_with_detections(det)

    # Q3 do Caio = canto inferior-esquerdo do frame (x < meio, y > meio)
    if TARGET_ID is None and len(det):
        for xyxy, tid in zip(det.xyxy, det.tracker_id):
            cx, cy = (xyxy[0]+xyxy[2])/2, (xyxy[1]+xyxy[3])/2
            if cx < info.width/2 and cy > info.height/2:
                TARGET_ID = int(tid); break

    mask = det.tracker_id == TARGET_ID if TARGET_ID is not None else np.zeros(len(det), bool)
    alvo = det[mask]
    out  = box.annotate(frame.copy(), alvo)
    out  = trace.annotate(out, alvo)
    return label.annotate(out, alvo, labels=[f"ALVO #{TARGET_ID}"]*len(alvo))

sv.process_video(source_path=SOURCE, target_path=TARGET, callback=annotate)
print("ok ->", TARGET)
